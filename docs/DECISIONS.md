# 决策记录

## 2026-06-08

### 决策

首版作品集使用无构建步骤的静态 HTML/CSS/JavaScript。

### 原因

- GitHub Pages 可以直接托管静态文件。
- 项目当前为空，静态实现上线最快，也便于后续其他 Agent 接手。
- 用户尚未提供真实作品数据，先用集中式数据结构降低替换成本。

### 影响

- 不需要 Node.js、Vite 或其他构建链。
- 页面交互保持轻量。
- 适合先上线，再逐步替换真实内容。

### 是否可逆

可逆。后续如果需要复杂路由、CMS 或动画系统，可以迁移到 React/Vite。

## 2026-06-08

### 决策

作品区优先从 GitHub API 读取 `LiJzd` 的公开仓库，并保留静态 fallback 数据。

### 原因

- 用户希望“我的作品都放进去”，公开 GitHub 仓库是当前最准确的作品来源。
- 动态读取可以覆盖后续新增公开仓库。
- fallback 可以避免 API 限流或网络异常时页面空白。

### 影响

- 首次加载作品区会请求 GitHub 公共接口。
- 静态 fallback 需要在重要仓库信息变化时手动同步。

### 是否可逆

可逆。后续可以改成完全静态数据，或引入构建时抓取仓库数据。

## 2026-06-08

### 决策

UI 参考 `LiJzd/ZUOPINGJI` 的视觉风格，但项目展示保持正常作品集表达。

### 原因

- 用户明确要求“只需要模仿风格和效果，项目展示还是按正常的来”。
- `ZUOPINGJI` 的黑红高对比、斜切、硬阴影和半调网点适合作为个人风格。
- 剧情化“劫案/预告信”等文案不适合当前作品集的信息展示目标。

### 影响

- 页面视觉更强烈，更接近参考仓库。
- 导航、项目卡片、弹窗和联系区仍使用项目/技能/关于/联系等常规信息架构。

### 是否可逆

可逆。可以回退到原白底作品集，也可以进一步迁移为 React/Vite 版本以复刻更多动态效果。
